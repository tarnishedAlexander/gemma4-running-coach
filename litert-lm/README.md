# litert-lm/ — Gemma 4 E2B on iPhone via LiteRT-LM

Parallel to the `llama-cpp/` implementation. Uses Google's **LiteRT-LM** runtime through the community Swift wrapper [`mylovelycodes/LiteRTLM-Swift`](https://github.com/mylovelycodes/LiteRTLM-Swift) — wraps LiteRT-LM's open-source C API with a clean async/await Swift surface.

## Why this exists

The `llama-cpp/` path works today and gives us full control, but it's GPU-only on Apple, doesn't use Multi-Token Prediction, and required ~150 lines of Swift bridging plus a custom XCFramework build to expose mtmd.

LiteRT-LM via this wrapper is the alternative path:
- ~4 lines of integration code instead of 150
- Audio + vision built-in (single API call each)
- MTP from v0.11+ baked in (>2× decode on mobile GPU)
- Same Gemma 4 E2B model, same iPhone target

Trade-off: depends on a community Swift package + prebuilt `CLiteRTLM.xcframework`. Less battle-tested than llama.cpp; one maintainer; rebuild from C source if/when needed.

## Model file

The 2.6 GB Gemma 4 E2B `.litertlm` model is **not in git** (exceeds GitHub's 100 MB per-file limit) and is **downloaded by the iOS app on first launch** via the bundled `ModelDownloader` (target: app's `Documents/` directory).

If you want to pre-stage it (e.g. to test the macOS LiteRT-LM CLI, or to seed an iOS Sim sandbox before launch):

```bash
mkdir -p ~/work/evan/gemma_models
curl -L -o ~/work/evan/gemma_models/gemma-4-E2B-it.litertlm \
  https://huggingface.co/litert-community/gemma-4-E2B-it-litert-lm/resolve/main/gemma-4-E2B-it.litertlm
```

Verify size after download — must be ~2.6 GB:
```bash
ls -lh ~/work/evan/gemma_models/gemma-4-E2B-it.litertlm
```

## Build

Requires Xcode 16+ and [XcodeGen](https://github.com/yonaskolb/XcodeGen). The repo includes a prebuilt XcodeGen binary in the apply.sh script (downloaded on demand if missing).

```bash
cd litert-lm
./apply.sh
```

That:
1. Downloads + installs XcodeGen locally if not on PATH (no sudo)
2. Runs `xcodegen generate` in `ios-app/` → produces `GemmaCoach.xcodeproj`
3. Resolves the `LiteRTLM-Swift` Swift Package dependency (pinned to `main` branch — no version tags published yet)
4. Builds for `generic/platform=iOS` (unsigned, Debug)
5. Outputs `ios-app/build_ios/Build/Products/Debug-iphoneos/GemmaCoach.app` (~40 MB, model NOT bundled)

## Deploy to physical iPhone

1. `open ios-app/GemmaCoach.xcodeproj`
2. Select the `GemmaCoach` target → Signing & Capabilities → tick **Automatically manage signing**, pick your Team
3. The bundle ID `com.gemma4.runningcoach.GemmaCoach` is unclaimed; if it collides, rename to anything unique
4. **Increased Memory Limit entitlement** is already configured (Gemma 4 needs ~4 GB RAM at load — without this iOS may kill the app)
5. Plug iPhone in (must be **iPhone 13 Pro or newer**, 6 GB+ RAM), select destination, ⌘R
6. On first launch the app downloads `gemma-4-E2B-it.litertlm` (~2.6 GB) from Hugging Face into the app's Documents directory. Allow it to finish (progress bar shown). Subsequent launches reuse the cached file.
7. Once status shows **Ready**:
   - Tap **Text only** for a text-only generation
   - Tap **Pick photo** → select an image → type a prompt → **Text + image** for vision
8. **Airplane mode:** safe AFTER first download. The model file is on-disk; engine init + inference are fully local.

## What's in the engine wrapper

`EngineModel.swift` is the single source of truth for LiteRT-LM access from SwiftUI. Public surface:

```swift
@MainActor final class EngineModel: ObservableObject {
    @Published var status: Status                 // idle/downloading/loading/ready/generating/error
    @Published var output: String
    @Published var lastTimeToFirstToken: Double
    @Published var lastDecodeTokensPerSecond: Double
    @Published var lastTotalTime: Double

    func loadIfNeeded() async                                      // download + load (idempotent)
    func generateText(_ prompt: String, maxTokens: Int) async      // streaming text
    func generateVision(imageData: Data, prompt: String, maxTokens: Int) async  // image + text
}
```

Wraps `LiteRTLMEngine.generateStreaming(...)` for text and `LiteRTLMEngine.vision(...)` for image input. The streaming path explicitly times TTFT from the first chunk; the vision path times total only (LiteRTLM's one-shot vision API doesn't expose token-level timing).

## Adding audio later

`engine.audio(audioData:prompt:format:)` is in the underlying API (WAV/FLAC/MP3, auto-resampled to 16 kHz mono). For a running coach this is the natural next step — feed in audio of breathing/footstrike/voice, get coaching back. Add a single method to `EngineModel` mirroring `generateVision` and a microphone button in `ContentView` and you're done.

`engine.multimodal(audioData:imagesData:prompt:)` for image + audio in one call.

## Performance reference (expected on iPhone 16 Pro)

Based on M4 LiteRT-LM CLI + MTP scaling (~3× slower on phone, but MTP gives 2× back):
- Text decode: **~30-40 tok/s** (MTP enabled, vs ~15-20 without)
- Vision: **~2-4 sec for 256-token image processing**
- Cold model load: **~5-10 sec** (mmap of 2.6 GB)

## When to prefer this over `../llama-cpp/`

- You want vision + audio + multimodal without writing C bridging code
- You want MTP (~2× decode speedup) for free
- You'd rather ship one Swift Package import than maintain a fork of llama.cpp + custom xcframework script
- Min device iPhone 13 Pro is acceptable (the llama-cpp path supports older devices)

## When to prefer `../llama-cpp/`

- You want absolute control over sampling, KV cache, custom samplers
- You don't trust prebuilt binaries from a small community repo
- You need to support pre–iPhone 13 devices
- Active community + huge surrounding ecosystem (Ollama, LM Studio, etc.)
